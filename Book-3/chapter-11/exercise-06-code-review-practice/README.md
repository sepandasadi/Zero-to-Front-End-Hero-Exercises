## üìã Overview

Code review is a critical skill for professional developers. This exercise teaches you how to review code effectively and provide constructive feedback.

## üîç What to Look For in Code Reviews

### **1. Code Quality**
- Clean code principles (DRY, KISS, SRP)
- Proper naming conventions
- Appropriate abstractions
- Code organization

### **2. Functionality**
- Does it work as intended?
- Edge cases handled?
- Error handling present?
- Tests included?

### **3. Performance**
- Unnecessary re-renders?
- Expensive operations optimized?
- Memory leaks prevented?

### **4. Security**
- Input validation?
- XSS prevention?
- Sensitive data protected?

### **5. Maintainability**
- Easy to understand?
- Well-documented?
- Consistent style?
- Future-proof?

## üìù Exercise Structure

You'll review **3 code samples** and provide feedback on:

1. **UserDashboard Component** - Has multiple quality issues
2. **API Module** - Has security and error handling issues
3. **Form Component** - Has performance and accessibility issues

For each, you'll:
- Identify issues
- Suggest improvements
- Provide constructive feedback
- Prioritize fixes

## ‚úÖ Code Review Checklist

**Before Approving:**
- [ ] Code follows project conventions
- [ ] No obvious bugs
- [ ] Tests pass (or included)
- [ ] No security vulnerabilities
- [ ] Performance acceptable
- [ ] Documentation updated
- [ ] No unnecessary complexity

## üí° Giving Good Feedback

### **DO:**
- Be specific and constructive
- Explain *why* something should change
- Suggest alternatives
- Praise good code
- Ask questions to understand intent

### **DON'T:**
- Be vague ("this is bad")
- Criticize the person
- Nitpick style if it matches project conventions
- Demand changes without explanation
